# NJ Energy Explorer

An interactive map of New Jersey electricity costs by ZIP code, layering wholesale
market prices (PJM day-ahead LMP) against retail residential bills (EIA-861M),
and adding a **modeled per-ZIP residential bill** built from Census/ACS housing
characteristics and calibrated against survey-reported bills.

Benchmark/reference: <https://electricity.heatmap.news/>.

```bash

# Steps 01–03 hit external APIs and are slow / rate-limited. The repo already
# ships a populated data/nj_lmp.db, so for a normal rebuild you only need:

python pipeline/01_pull_lmp.py           # needs PJM API key (in script)

python pipeline/03_pull_acs.py           # needs CENSUS_API_KEY in .env

python -m http.server 8000 --directory ui   # then visit http://localhost:8000
```

`03_pull_acs.py` needs a free Census API key:
<https://api.census.gov/data/key_signup.html>, placed in `.env` as
`CENSUS_API_KEY=...`.

## 3. Methodology

The project answers two questions at the ZIP level: *what does electricity cost
on the wholesale market here* (LMP), and *what does a residential household pay*
(the bill). Wholesale and retail are separate worlds — the methodology layers
them and then adds a modeled bill that varies **within** a utility, which neither
the raw EIA data nor the reference site does.

### 3.1 Wholesale price — PJM day-ahead LMP (`01_pull_lmp.py`)

- **Source:** PJM Data Miner 2 API, day-ahead hourly LMPs (`da_hrl_lmps`).
- **What LMP is:** the Locational Marginal Price is the wholesale cost of
  delivering 1 MWh of electricity to a specific grid node — generation cost plus
  congestion plus losses. It is *not* what a household pays; it is the wholesale
  input that utilities buy at.
- **Method:** for the 1st of every month, 2020–present, all 24 hourly LMPs at
  every NJ load node (PSEG, JCPL, AECO, RECO zones) are pulled and averaged to a
  daily value. Each ZIP is mapped to its nearest PJM node by haversine distance.
- **Stored in:** `nj_lmp.db` → tables `zips` (one row per ZIP: node, zone,
  distance) and `lmp_daily` (one row per node-date).

### 3.2 Retail bill — EIA-861M (`02_pull_eia861m.py`)

- **Source:** EIA-861M monthly survey of utility retail sales
  (`sales_ult_cust_<year>.xlsx`), residential sector.
- **Scope:** the 4 NJ investor-owned utilities that issue residential bills —
  Atlantic City Electric (EIA id 963), JCP&L (9726), PSE&G (15477), Rockland
  Electric (16213).
- **Method:** for each utility-month we compute
  - `rate_cents_per_kwh = revenue / sales × 100`
  - `bill_dollars       = revenue / customer_count`
  - `kwh_per_customer   = sales / customer_count`
- **Key limitation:** EIA reports one number per **utility**, so every ZIP in a
  service territory shares the same rate and bill. The reference site
  (heatmap.news) has the same limitation — its map only changes at utility
  boundaries. This is the gap the modeled bill (§3.4) closes.
- **ZIP → utility mapping:** NJ utility territories roughly follow county lines,
  so ZIPs are assigned by county, with a hand-coded override for the small
  Rockland Electric pocket in far-north Bergen County (Mahwah–Ramsey corridor).
  Stored in `nj_lmp.db` → `utility_monthly`.

### 3.3 Housing characteristics — Census ACS (`03_pull_acs.py`)

- **Source:** Census ACS 5-year (2023 vintage) via the Census API, for every NJ
  ZCTA. Two tables:
  - **B25024 — Units in Structure** (detached single-family, attached, 2-unit,
    3–4, 5–9, 10–19, 20–49, 50+, mobile, other). The *housing density* lever.
  - **B25018 — Median rooms per unit.** The *home size* lever.
- **Cleaning:** Census uses large negative sentinels (`-666666666`, …) for
  suppressed estimates; these are scrubbed to null before storage so they don't
  poison averages.
- **Coverage:** 592 of 598 NJ ZCTAs have usable housing data (the rest are
  PO-box-only ZIPs). Stored in `nj_lmp.db` → `acs_zcta`.

### 3.4 Modeled per-ZIP bill (`05_build_ui_json.py`)

The core idea: take the utility-wide EIA bill and **split it across ZIPs** by how
much electricity each ZIP's housing stock would actually use, then **calibrate
the level** against survey-reported bills.

**Step A — consumption index.** For each ZIP, build a unitless multiplier
`c_zip = housing_mix_factor × size_factor`:

- *housing_mix_factor* weights each B25024 structure type by its relative
  electricity use, anchored to EIA RECS Middle-Atlantic kWh-per-housing-type
  ratios (detached single-family = 1.00, attached = 0.85, 2-unit = 0.78, down to
  50+ unit buildings = 0.50). A ZIP of high-rise apartments lands near 0.55; a
  leafy detached-SFH ZIP near 1.0.
- *size_factor* adds ±5% per room above/below the utility's median rooms, clipped
  to ±30% so a single outlier ZIP can't dominate.

The index is then **renormalized within each utility** so the housing-unit
weighted mean = 1.0. This preserves the EIA-861M total: we *redistribute* a
utility's bill across its ZIPs, we don't invent new aggregate dollars.

**Step B — census calibration.** EIA's `revenue / customers` systematically
under-states the per-household bill (a household can sit behind multiple billing
accounts), so the raw modeled bills ran ~$69/month below the Census ground truth.
We apply a **per-utility multiplicative level shift** equal to
`census_mean / modeled_mean` over the overlap years (2021–2024). This corrects
the level while leaving the within-utility shape (the part the ACS index
produces) untouched. The fitted factors:

| Utility                 | calibration factor |
|-------------------------|--------------------|
| PSE&G                   | 1.737              |
| Atlantic City Electric  | 1.446              |
| JCP&L                   | 1.432              |
| Rockland Electric       | 1.304              |

PSE&G needs the largest lift, consistent with dense urban service territories
having the most multi-account households per dwelling.

**Output:** modeled bill per ZIP-month = `EIA utility bill × c_zip × calibration`.
Written into `ui/nj_zip_info.json` (per-ZIP series) and exported flat to
`data/outputs/nj_zip_prices.txt`.

### 3.5 Validation against Census ground truth (`06_validate_vs_census.py`)

- **Ground truth:** `data/raw/census_acs_electricity_bills.csv` — average monthly
  residential electricity bill per NJ ZIP per year, 2014–2024 (2020 missing),
  derived from ACS household-reported expenditure. This is the same family of
  measurement used in the reference data-center paper, and is the right yardstick
  because it is a *measured* household bill, not an accounting figure.
- **Method:** our monthly modeled bills are aggregated to an annual mean per ZIP
  and merged with the census bills on the overlapping years (2021–2024); we
  report Pearson/Spearman correlation, bias, MAE, and MAPE — pooled, per year,
  and per utility.

### 3.6 The quantities we measure — units and why they matter

The map carries four distinct dollar/energy measures. They are easy to confuse
because they all describe "the price of electricity," but they sit at different
points in the supply chain and answer different questions. Getting the units
right is the whole point — a number is meaningless without knowing whether it is
wholesale or retail, per-MWh or per-month.

**Day-ahead LMP — `$/MWh` (wholesale).**
The Locational Marginal Price is what it costs, on the wholesale market, to
deliver one megawatt-hour to a specific point on the grid one day ahead of
real time. It decomposes into three parts: the marginal generator's energy cost,
the cost of grid congestion getting power to that location, and line losses.
- *Why it matters:* LMP is the upstream signal — it is what utilities and large
  buyers actually pay into the market, and it is where the effect of new large
  loads (like data centers) would first show up. It is volatile (it moves hour to
  hour) and it is **not** what a household pays. A household never sees $/MWh;
  they see it only indirectly, months later, after the utility blends it into a
  regulated retail rate. Tracking LMP separately lets us ask "did wholesale costs
  move?" independently of "did bills move?".
- *Unit note:* 1 MWh = 1,000 kWh, so an LMP of $40/MWh is 4¢/kWh of pure energy
  cost — before transmission, distribution, taxes, and utility margin are added.

**Residential rate — `¢/kWh` (retail, per unit).**
Total residential revenue divided by total residential kWh sold, for a utility in
a month (`revenue / sales × 100`). This is the *all-in* retail price of a unit of
electricity: generation + transmission + distribution + fees, averaged across all
residential customers.
- *Why it matters:* the rate is the cleanest apples-to-apples comparison **between
  utilities** because it is normalized per kWh — it strips out the fact that some
  households simply use more electricity than others. If you want to know "is
  PSE&G more expensive than JCP&L per unit," this is the number. It is typically
  2–3× the underlying LMP, and the gap is the cost of delivery and regulation.

**Residential bill — `$/month` (retail, per household).**
Total residential revenue divided by the number of residential customer accounts
(`revenue / customers`). This is the dollar figure a typical household sees.
- *Why it matters:* the bill is what people actually experience and what policy
  debates are about. It is `rate × quantity`, so it blends *price* (the rate) with
  *consumption* (how much a household uses). A ZIP can have a high bill either
  because the rate is high or because the homes are large and use a lot — the bill
  alone can't tell you which. That ambiguity is exactly why we model it (§3.4) and
  why the consumption index below exists.
- *Caveat:* "customers" counts billing **accounts**, not households or people. A
  household behind several meters, or a master-metered apartment building, distorts
  this number — which is the source of the level bias the calibration corrects
  (§3.4 Step B).

**kWh per customer — `kWh/month` (consumption).**
Total residential kWh divided by customer accounts (`sales / customers`). The
average monthly electricity *consumption* per account, with price removed.
- *Why it matters:* this is the consumption half of the bill. It is what physically
  differs between a one-bedroom apartment and a four-bedroom detached house, and it
  is the quantity our ACS housing-mix index is trying to approximate at the ZIP
  level. Comparing kWh/customer across utilities shows usage differences;
  comparing it across months shows seasonality (summer AC, winter heat).

**Consumption index — unitless multiplier (the model's core).**
A relative number centered on 1.0 that says how a ZIP's expected consumption
compares to its utility's average: 1.0 = average household, 1.3 = uses ~30% more,
0.7 = uses ~30% less. Built from housing mix and home size (§3.4 Step A).
- *Why it matters:* it is deliberately **unitless and utility-normalized** so it
  can redistribute a utility's bill across ZIPs without inventing new dollars.
  Multiplying a single utility-wide bill by each ZIP's index is what turns one flat
  number into a varying surface. The index is the bridge between *what a utility
  charges on average* (EIA) and *what a particular neighborhood's housing stock
  would consume* (ACS).

## 4. Results

Fit of the modeled bill against Census ground truth (2021–2024, n = 2,298
ZIP-years):

| Metric            | Before calibration | After calibration |
|-------------------|--------------------|-------------------|
| Pearson r         | 0.420              | **0.527**         |
| Spearman r        | 0.420              | **0.536**         |
| Mean bias         | −$68.70            | **$0.00**         |
| MAE               | $68.72             | **$26.06**        |
| MAPE              | 35.1%              | **13.9%**         |

The per-utility level shift removes the entire $69 bias and cuts mean absolute
error to ~$26/month (14% MAPE). The remaining correlation (~0.53) is the honest
ceiling for this approach: the census ground truth is itself smoothed across
sub-ZIP geographies, so genuine within-area variation that our housing-mix index
captures is partly scored as "error" against a smoothed reference.

### 4.1 How to read the fit metrics

Each statistic answers a different question about how well the modeled bills match
the Census ground truth. They are reported together because no single one is
sufficient — a model can rank ZIPs perfectly yet be biased by $50, or be unbiased
on average yet wrong on every individual ZIP. Below, *modeled* = our value,
*census* = ground truth, *n* = number of ZIP-years compared.

**Pearson correlation (r) — "do the two move together, linearly?"**
`r = cov(modeled, census) / (σ_modeled · σ_census)`, ranging −1 to +1. It measures
how tightly the points fall on a straight line. r = 1 is a perfect linear match,
0 is no linear relationship.
- *Why it matters:* this is the headline "does the model capture the shape of the
  variation" metric. Crucially, **r is immune to level and scale** — adding $50 to
  every modeled bill, or multiplying them all by 1.7, leaves r unchanged. That is
  exactly why r barely cares about the calibration step (it went from 0.42 to 0.53
  only because re-weighting interacts slightly with which ZIPs overlap), and why we
  *also* report bias and MAE, which calibration does fix.
- *Reading ours:* ~0.53 means the model explains roughly r² ≈ 28% of the
  cross-ZIP variance in bills. Moderate — good enough to be informative, limited by
  the smoothed reference and by the fact that we model only housing structure, not
  behavior, income, or heating fuel.

**Spearman correlation (ρ) — "do the two agree on the ranking?"**
The same Pearson formula applied to the *ranks* of the values instead of the raw
values. It asks whether the model puts ZIPs in the right order (cheapest to most
expensive), regardless of the exact spacing.
- *Why it matters:* it is robust to outliers and to non-linear-but-monotonic
  relationships. If our model systematically compresses or stretches the high end
  but still orders ZIPs correctly, Spearman stays high while Pearson drops.
  Comparing the two diagnoses the *type* of error: here Pearson (0.527) and
  Spearman (0.536) are nearly equal, which tells us the relationship is essentially
  linear with no major rank-vs-magnitude discrepancy.

**Mean bias — "are we systematically too high or too low?"**
`mean(modeled − census)`, in dollars. The average signed error. Positive = we
over-predict, negative = we under-predict.
- *Why it matters:* bias is the one error a *level* calibration can eliminate
  completely, and it is the most visible failure to a non-technical viewer — if
  every bill on the map reads $69 too low, the map is simply wrong even if the
  pattern is right. Pre-calibration bias was −$68.70 (we under-predicted by ~$69
  everywhere because EIA's revenue/customer under-states the household bill).
  Post-calibration it is $0.00 **by construction** — the per-utility factor is
  defined as `census_mean / modeled_mean`, which forces the means to match. So a
  $0 bias is not evidence the model is good; it is evidence the calibration did its
  one job. The metrics that show real quality are the next two.

**MAE — Mean Absolute Error — "how far off is a typical ZIP, in dollars?"**
`mean(|modeled − census|)`. Unlike bias, it does not let positive and negative
errors cancel, so it reflects the typical magnitude of error regardless of
direction.
- *Why it matters:* this is the most intuitive accuracy number — "on average we are
  off by $X per month." It is in the original units (dollars), so it is directly
  meaningful. It fell from $68.72 to $26.06 with calibration: once the level bias
  is removed, the average remaining miss on any given ZIP is about $26/month. MAE
  treats a $10 miss and a $50 miss proportionally (unlike RMSE, which would
  penalize the $50 miss far more) — we use MAE because we care about typical error,
  not about being dominated by a few extreme ZIPs.

**MAPE — Mean Absolute Percentage Error — "how far off in relative terms?"**
`mean(|modeled − census| / census) × 100`, a percentage.
- *Why it matters:* MAE in dollars can look large or small depending on the
  baseline — $26 off a $190 bill is very different from $26 off a $40 bill. MAPE
  normalizes by the true value so errors are comparable across high- and low-bill
  ZIPs, and it is the most portable way to state accuracy ("within ~14%"). Our 13.9%
  means a typical modeled bill lands within about one-seventh of the true value —
  reasonable for a structural model that uses no household-level data.
- *Caveat:* MAPE is asymmetric and blows up when the denominator is small; it is
  safe here because residential bills are never near zero.

**Why pooled, per-year, and per-utility.**
The same metrics are reported three ways. *Pooled* gives the overall headline.
*Per-year* checks whether the fit is stable over time or driven by one good year
(ours holds ~0.46–0.59 across 2021–2024). *Per-utility* checks where the model
works best — the calibration forces per-utility bias to ~$0 by construction, so the
informative per-utility number is the correlation, which reveals that the model
tracks within-territory variation better in some service areas (e.g. Rockland
r ≈ 0.66) than others (JCP&L r ≈ 0.27, a sprawling territory mixing dense shore
towns with rural northwest NJ).

**Example — within PSE&G (same utility-average bill, different housing stock):**

| ZIP   | Area              | % detached SFH | modeled bill |
|-------|-------------------|----------------|--------------|
| 07310 | Jersey City       | ~0% (high-rise)| ~$72/mo (pre-cal) |
| 07021 | Essex Fells       | 92%            | ~$202/mo (pre-cal) |

Before this work every PSE&G ZIP showed an identical bill. The model now produces
plausible geographic structure that tracks housing density and home size.
